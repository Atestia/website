The BCBS 239 lineage gap nobody is solving.

Sean Kuesia · Published 2026-06-02

The Basel Committee on Banking Supervision published BCBS 239 in January 2013. The document — formally titled "Principles for effective risk data aggregation and risk reporting" — establishes fourteen principles that globally systemically important banks (G-SIBs) and domestic systemically important banks (D-SIBs) must implement. The original compliance deadline for G-SIBs was January 2016. Ten years later, most banks remain partially compliant.

The reason is mostly architectural. BCBS 239 demands end-to-end lineage from source system to risk report. Banks have spent the last decade buying data catalogs — Collibra, Alation, Informatica — that document data assets but do not capture lineage events at runtime. The catalog approach works for stable schemas where lineage can be documented once and updated quarterly. It does not work for the modern AI-augmented data pipeline where transformations are spawned dynamically by Spark jobs, dbt models, and agentic workflows.

The fourteen principles, summarized

BCBS 239's principles fall into three tiers. The first tier is governance and infrastructure: a bank's data architecture and IT infrastructure must support effective risk data aggregation, with strong governance and clear ownership. The second tier is data aggregation capabilities: accuracy and integrity, completeness, timeliness, adaptability. The third tier is reporting practices: accuracy, comprehensiveness, clarity, frequency, and distribution.

Every principle assumes the bank can answer "where did this number come from?" — for any number that appears in a regulatory report, the bank must be able to trace the data back to source systems, document the transformations, and prove the lineage held under audit.

This is the lineage requirement. It is the foundation under everything else BCBS 239 asks for.

What the data catalog approach captures and misses

A modern data catalog records the static structure of the data estate: which tables exist, which columns they contain, what types, which downstream tables they feed. Modern catalogs use automated parsers (Atlan's reverse SQL, Alation's Compose, Collibra's dataLineage) to extract column-level lineage from SQL statements. This is genuinely useful for impact analysis, data discovery, and basic compliance attestation.

It misses three things.

One: dynamic transformations. A Spark job that joins two tables and writes a third does not necessarily appear in the catalog until the next scan. If the scan runs nightly and the job ran six hours ago to populate a risk report, the catalog's lineage view is incorrect.

Two: agentic invocations. When a bank's risk data pipeline now includes calls to vendor LLM models — for example, an agent that classifies counterparty relationships, summarizes news, or produces narrative explanations for risk reports — the catalog cannot represent that. The catalog has no concept of "an LLM saw this data and produced this output."

Three: temporal lineage. When an examiner asks "what was the lineage of this CCAR number as of June 30, 2026," the catalog's current view does not answer the question. Banks need point-in-time lineage, not catalog state.

The runtime lineage layer

The architectural answer is to capture lineage events at the point where data flows happen — runtime — rather than reconstructing lineage from static structure. Every Spark job emits an event. Every dbt run emits an event. Every Airflow DAG execution emits an event. Every agentic LLM invocation emits an event. These events are persisted with timestamps and form an append-only log of what actually happened.

The OpenLineage specification, maintained by the Linux Foundation AI & Data Foundation, defines the canonical event format. It is implemented natively by Airflow, dbt, Snowflake (via Horizon), and Databricks (via Unity Catalog). Apache Spark has an OpenLineage integration. Marquez is the open-source reference implementation of the OpenLineage backend.

This is the missing layer in most banks' BCBS 239 implementations. Banks have catalogs. Banks have lineage parsers. Banks do not have OpenLineage emission at runtime across their data estate.

Where Atestia fits

The Atestia Tier-4 MCP server adds two things to the OpenLineage runtime lineage story.

First, it adds vendor model invocations as a first-class lineage event type. When a bank's agentic AI calls Experian for a credit check, calls Plaid for an account lookup, or calls Claude for narrative generation, that call becomes a Tier-4 lineage record. The lineage record links to upstream OpenLineage data lineage events (the source tables the agent read from) and downstream OpenLineage events (the tables the agent wrote to). Vendor model invocations become navigable nodes in the lineage graph.

Second, it adds the BCBS 239 packet generator. Given a target regulatory report — CCAR Y-14A schedule, DFAST submission, Y-15 quarterly call — the packet generator walks backwards through the lineage graph and produces a BCBS 239-conformant aggregation reference document. The document maps every data element in the report back through the transformation chain to source systems, with timestamps, transformation rationale, and validation links.

The reverse pattern

The pattern also runs in reverse. When an OCC examiner walks into a bank's MRM team and asks "show me the BCBS 239 evidence for this CCAR number that was reported nine months ago," the MRM team can issue a query against the Atestia lineage substrate. The substrate returns the lineage graph as of the report submission date. The graph identifies every source system, every transformation, every vendor model invocation, and every validation event that contributed to the number. The team produces a packet.

This is the workflow banks have wanted since 2013 but could not build with catalog tools alone. The runtime lineage layer is what makes it possible.

If you already have OpenLineage emission

Most banks that have invested in OpenLineage emission did so to feed Marquez or to integrate with Snowflake Horizon. The Atestia MCP server consumes the same OpenLineage event stream and joins it to vendor model invocation events that OpenLineage cannot represent natively. The integration is additive, not replacement. You keep your existing OpenLineage backend.

If you have not invested in OpenLineage emission yet, starting with the Atestia MCP server gives you the vendor-model side of the lineage graph immediately, and you can backfill data-side OpenLineage emission incrementally — Spark today, dbt next quarter, Airflow when convenient.

What to do this week

If you are running BCBS 239 for a G-SIB or D-SIB and you have not yet stood up runtime lineage capture, the practical path is:

  • Inventory your existing OpenLineage emission. Most banks have more than they realize. Snowflake Horizon, Databricks Unity Catalog, Airflow on EKS, and most dbt deployments emit OpenLineage by default.
  • Install @atestia/tier4-mcp-server in a sandbox. Configure it as an MCP server in your AI agent stack. Capture vendor model invocations for one workflow.
  • Try tier4.packet.bcbs-239 for a target regulatory report. Compare the generated packet to what your team manually assembles today.
