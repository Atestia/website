What is actually inside a Tier-4 attestation packet.

Sean Kuesia · Published 2026-06-02

The Atestia Tier-4 MCP server ships with five packet generators. Each generator produces a structured artifact intended for a specific regulatory or operational consumer: examiners, MRM analysts, auditors, customers, or downstream OpenLineage backends. This post walks through the actual contents of each packet, why each field exists, and what the consumer is supposed to do with it.

Lineage records: the underlying primitive

Every packet is built from one or more lineage records. A lineage record is the canonical evidence object Atestia captures at the moment a vendor model is invoked. A complete lineage record contains:

{
  "lineageId": "tier4-ln-eCVNqVHlyh9bub-ENNsLvg",
  "vendorAttestationUri": "tier4://vendor/experian-business-credit-v3.2",
  "sr26_2PacketUri": "tier4://packet/sr-26-2/experian-business-credit-v3.2",
  "bcbs239AggregationId": "tier4-bcbs239-vziGaXf-Rnw",
  "tenantId": "citi-arc-mrm-tenant",
  "agentId": "citi-arc-agent-demo-001",
  "invocationId": "inv-demo-001",
  "vendorModelId": "experian-business-credit-v3.2",
  "vendorModelVersion": "v3.2.7",
  "inputPayloadHash": "a1b2c3...",
  "inputSummary": "SMB credit check; Acme Corp; $500K facility",
  "decisionContext": "Tier-2 expansion line; 18-month term",
  "invokedAt": "2026-06-02T14:23:17.523Z",
  "latencyMs": 384,
  "succeeded": true,
  "errorMessage": null,
  "specVersion": "1.2"
}

Three things to notice. First, the input is hashed, never stored raw. Atestia is provably PII-free. Second, the URI fields are forward references to the packets that can be generated from this record. Third, the spec version is part of the record, so packet generation is reproducible against historical schema definitions.

OCC 2013-29 vendor questionnaire

OCC Bulletin 2013-29 ("Third-Party Relationships: Risk Management Guidance") established the questionnaire pattern that bank vendor risk teams have used since 2013. The questionnaire is 47 fields across 7 sections covering vendor due diligence, contract review, ongoing monitoring, and termination.

The Atestia packet generator produces a JSON document that matches the section structure with three layers of content for each field: (a) the value derived from the lineage record and the vendor registry, (b) the source citation explaining where the value came from, and (c) the supporting lineage record IDs that substantiate the value.

A representative snippet:

{
  "packetType": "occ-2013-29",
  "packetId": "tier4-occ-Rs9j10mU72Y",
  "vendorModelId": "experian-business-credit-v3.2",
  "section_1_vendor_due_diligence": {
    "1_1_vendor_legal_name": {
      "value": "Experian Information Solutions, Inc.",
      "source": "vendor_registry.json",
      "lastVerifiedAt": "2026-05-15T00:00:00Z"
    },
    "1_2_service_description": {
      "value": "Business credit scoring API",
      "source": "vendor_contract:msa-2024-experian.pdf",
      "supportingLineage": ["tier4-ln-eCVNqVHlyh9bub-ENNsLvg", "..."]
    },
    ...
  }
}

The packet is consumable by examiners who want machine-readable evidence and by MRM analysts who want a workflow tool input. The format is documented in the spec section 5.

SR 26-2 attestation pack

The SR 26-2 attestation pack is the bundle: a 5-artifact set generated together for the OCC examiner cycle. The artifacts are:

  • Model card. Vendor model identification, version, training methodology summary, intended use, fairness considerations. Generated from the vendor registry.
  • OCC questionnaire. The full 47-field document described above.
  • Lineage trail spec. A summary of the lineage records captured for invocations of this vendor model during the reporting period, with statistical aggregates (invocation count, p50/p95/p99 latency, success rate).
  • Monitoring spec. How the bank monitors this vendor model in production: SLO thresholds, drift detection signals, escalation procedures.
  • SOC 2 control matrix. A crosswalk from the vendor's SOC 2 Type 2 controls to the bank's MRM control framework, with gap analysis.

The pack is delivered as a ZIP or as five separate JSON documents. The recommended consumer workflow is for the MRM team to review the pack quarterly and refresh artifacts annually.

BCBS 239 aggregation reference

The BCBS 239 aggregation reference is the data lineage packet. It maps a target regulatory report — CCAR Y-14A, DFAST submission, FFIEC 031, FR Y-15 — to source systems through every transformation in between. The packet includes:

  • Target report identifier and submission date.
  • For each data element in the report: source system, source table/column, transformation chain (Spark jobs, dbt models, Snowflake views), and validation steps.
  • Tier-4 lineage record IDs that substantiate each step.
  • OpenLineage event references for the data engineering side of the pipeline.
  • BCBS 239 principle conformance attestation for the 14 principles applicable to the report.

The packet is the answer to the examiner question "where did this number come from?"

ECOA Reg B Adverse Action Notice

Regulation B implements the Equal Credit Opportunity Act. Section 1002.9 requires that creditors provide adverse action notices when they take adverse action on a credit application. Section 1002.12 requires that creditors retain compliance records for 25 months.

The Atestia packet generator produces both the customer-facing notice and the internal compliance record from a single invocation:

{
  "packetType": "regb-adverse-action-notice",
  "packetId": "tier4-regb-Rs9j10mU72Y",
  "validUntil": "2028-07-02T...",  // 25 months out
  "customerNotice": {
    "applicantName": "Acme Corp",
    "actionType": "denial",
    "principalReasons": [
      "Insufficient business credit score",
      "Length of business operating history",
      "Existing debt-to-revenue ratio"
    ],
    "enforcementContact": {
      "agency": "CFPB",
      "address": "1700 G St NW, Washington DC 20552",
      "phone": "855-411-2372"
    },
    "noticeText": "..."
  },
  "complianceRecord": {
    "vendorModelsInvoked": ["experian-business-credit-v3.2"],
    "lineageRecordIds": ["tier4-ln-..."],
    "reasoningCaptureId": "tier4-rea-...",
    "humanReviewerSignature": "loan-officer-alice-zhang",
    "humanReviewOutcome": "approved",
    "retentionExpiresAt": "2028-07-02T..."
  }
}

The customer notice is the document the bank mails or emails to the applicant. The compliance record is what the bank keeps in its files for the 25-month CFPB retention window. Both are produced from one call.

OpenLineage RunEvent export

The final packet type is not regulatory — it is ecosystem integration. Atestia lineage records can be exported as OpenLineage RunEvent format and forwarded to existing OpenLineage backends (Marquez, Snowflake Horizon, Databricks Unity Catalog, Astronomer Cloud).

This is how bank data teams unify the vendor-model side of the lineage graph (captured by Atestia) with the data engineering side (captured by Airflow, dbt, Spark, Snowflake). The unified view is what BCBS 239 has wanted for ten years.
